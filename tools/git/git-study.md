\# Git 常用命令、分支策略与 Merge / Rebase 对比



\## 一、常用核心命令



\### 1. 基础操作

\- \*\*git init\*\*：初始化本地 Git 仓库

\- \*\*git clone \[url]\*\*：克隆远程仓库到本地

\- \*\*git add .\*\*：暂存所有修改文件至暂存区

\- \*\*git commit -m "描述"\*\*：提交暂存区代码到本地仓库，备注提交信息

\- \*\*git status\*\*：查看当前文件状态、暂存情况

\- \*\*git log\*\*：查看本地提交日志，`git log --oneline` 精简展示



\### 2. 远程交互

\- \*\*git remote add origin \[url]\*\*：绑定远程仓库地址

\- \*\*git pull\*\*：拉取远程最新代码并合并到本地当前分支

\- \*\*git push origin \[分支名]\*\*：推送本地分支代码到远程对应分支

\- \*\*git remote -v\*\*：查看远程仓库绑定信息



\### 3. 分支基础操作

\- \*\*git branch\*\*：查看本地所有分支

\- \*\*git branch \[新分支名]\*\*：新建分支，不切换

\- \*\*git checkout \[分支名]\*\*：切换指定分支

\- \*\*git checkout -b \[新分支名]\*\*：新建并立即切换到新分支

\- \*\*git branch -d \[分支名]\*\*：删除已合并的本地分支

\- \*\*git push origin --delete \[分支名]\*\*：删除远程分支



\### 4. 代码回退与撤销

\- \*\*git reset --hard \[commitId]\*\*：硬回退到指定提交，丢弃后续所有修改

\- \*\*git checkout .\*\*：撤销工作区所有未暂存的修改

\- \*\*git stash\*\*：暂存当前未提交的修改，清空工作区

\- \*\*git stash pop\*\*：恢复最近一次暂存的代码并删除暂存记录



\## 二、标准团队分支策略（企业通用）



采用主干开发、分支迭代模式，区分环境分支，保障代码稳定性，适配团队协作与版本发布。



\### 1. 核心常驻分支

\- \*\*main / master\*\*：生产环境主干分支，仅存放稳定上线代码，禁止直接提交，只能通过合并并入

\- \*\*develop\*\*：开发环境主干分支，聚合所有开发功能，作为迭代测试基础分支



\### 2. 临时迭代分支

\- \*\*feature/\*\*\*：功能开发分支，从 develop 拉出，单功能独立开发，完成后合并回 develop

\- \*\*bugfix/\*\*\*：测试 Bug 修复分支，从 develop 拉出，修复完成合并回 develop

\- \*\*hotfix/\*\*\*：线上紧急补丁分支，从 main 拉出，修复线上问题后，同时合并到 main 和 develop



\### 3. 协作规范

\- 所有功能、修复均在独立临时分支开发，杜绝直接操作主干分支

\- 分支开发完成后通过 PR/MR 合并，经过代码评审后并入目标分支

\- 合并完成后及时删除无用临时分支，保持仓库整洁



\## 三、Merge 与 Rebase 核心对比



\### 1. Git Merge（合并）

将指定分支的代码完整合并到当前分支，保留所有提交记录，会生成一条新的「合并提交记录」。



\*\*优点\*\*

\- 操作简单、风险极低，不修改原有提交历史

\- 完整保留分支开发轨迹，溯源清晰，适合公共主干分支



\*\*缺点\*\*

\- 迭代频繁时会产生大量合并节点，分支历史杂乱

\- 主干分支提交记录冗余，可读性差



\*\*适用场景\*\*

公共主干分支（main/develop）、多人协作公共分支合并，优先保证历史可追溯。



\### 2. Git Rebase（变基）

将当前分支的所有提交记录，平移复刻到目标分支最新提交节点之上，改写提交历史，形成线性干净的提交树。



\*\*优点\*\*

\- 提交历史线性整洁，无多余合并节点，可读性极强

\- 避免主干分支历史臃肿，适配持续迭代开发



\*\*缺点\*\*

\- 改写原有提交历史，绝对禁止在公共分支使用

\- 冲突解决集中，操作不当易造成代码覆盖、版本混乱



\*\*适用场景\*\*

个人本地功能分支、私有分支，用于同步主干最新代码，整理个人提交历史。



\### 3. 核心取舍原则（团队通用规范）

\- \*\*公共分支只用 Merge\*\*：保护公共提交历史，避免多人协作历史错乱

\- \*\*个人分支优先 Rebase\*\*：同步主干代码、整理本地提交，保证合并后主干历史干净
