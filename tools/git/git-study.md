# Git 常用命令、分支策略与 Merge / Rebase 对比

## 一、常用核心命令
### 1. 基础操作
- **git init**：初始化本地 Git 仓库
- **git clone [url]**：克隆远程仓库至本地
- **git add .**：将全部修改文件加入暂存区
- **git commit -m "描述"**：提交暂存区代码到本地仓库，附带修改说明
- **git status**：查看文件变更、暂存状态
- **git log**：查看提交记录；`git log --oneline` 精简单行展示日志

### 2. 远程仓库交互
- **git remote add origin [url]**：绑定远程仓库地址
- **git pull**：拉取远程最新代码并合并至当前分支
- **git push origin [分支名]**：推送本地分支至对应远程分支
- **git remote -v**：查看已绑定的远程仓库地址

### 3. 分支管理操作
- **git branch**：查看本地所有分支
- **git branch [新分支名]**：新建分支，不切换
- **git checkout [分支名]**：切换至指定分支
- **git checkout -b [新分支名]**：新建并直接切换到新分支
- **git branch -d [分支名]**：删除已完成合并的本地分支
- **git push origin --delete [分支名]**：删除远程分支

### 4. 代码回退与临时保存
- **git reset --hard [commitId]**：强制回退至指定提交，丢弃后续全部修改
- **git checkout .**：撤销工作区未暂存的全部改动
- **git stash**：临时存放未提交修改，清空工作区
- **git stash pop**：恢复最近一次暂存代码，并清除暂存记录

## 二、企业标准团队分支策略
采用「主干常驻 + 临时分支迭代」模式，隔离生产、开发环境，保障版本稳定，适配多人协作发布。

### 1. 常驻核心分支
- **main / master**：生产主干，仅存放上线稳定代码，禁止直接提交，仅通过合并流入
- **develop**：开发主干，汇总所有待迭代功能，作为测试、开发统一基准

### 2. 临时功能分支（用完即删）
- **feature/***：新功能分支，从 develop 拉出，开发完成后合并回 develop
- **bugfix/***：测试缺陷修复分支，从 develop 拉出，修复后合并回 develop
- **hotfix/***：线上紧急补丁分支，从 main 拉出，修复后同时合并至 main、develop

### 3. 团队协作规范
1. 所有功能、缺陷修复均在独立临时分支开发，禁止直接修改主干；
2. 分支开发完成后提交 PR/MR，经代码评审再合并至目标分支；
3. 合并完成后及时清理废弃临时分支，简化仓库分支列表。

## 三、Merge 与 Rebase 核心对比
### 1. Git Merge（合并）
将目标分支完整并入当前分支，保留完整开发历史，自动生成一条独立的合并提交记录。
#### 优点
- 操作门槛低、风险小，不会改写原有提交历史；
- 完整留存分支开发轨迹，问题溯源清晰，适合公共主干。
#### 缺点
- 频繁迭代会产生大量合并节点，提交树杂乱臃肿；
- 主干日志冗余，阅读成本高。
#### 适用场景
多人共用公共主干（main/develop）合并，优先保证提交历史不可篡改、可追溯。

### 2. Git Rebase（变基）
将当前分支全部提交平移复刻到目标分支最新节点，改写提交历史，生成一条线性干净的提交链。
#### 优点
- 提交记录线性整洁，无多余合并节点，可读性强；
- 减少主干冗余提交，便于持续迭代维护。
#### 缺点
- 会改写提交历史，**严禁在多人共用的公共分支执行**；
- 冲突集中处理，操作失误易引发代码覆盖、版本错乱。
#### 适用场景
个人私有功能分支：同步主干最新代码、整理本地零散提交，合并后保证主干历史整洁。

### 3. 团队统一取舍规范
1. 公共主干分支仅使用 Merge，保护全团队共享提交历史；
2. 个人开发分支优先使用 Rebase，同步主干、整理本地提交，降低主干历史杂乱度。
