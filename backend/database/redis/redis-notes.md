# Redis 完整核心知识点与面试八股笔记

## 1. Redis 五大基础数据类型、底层结构与业务用途

### 1.1 String 字符串
- 底层：SDS 简单动态字符串
- 用途：普通K-V缓存、计数器、分布式Session、验证码存储
- 常用命令：set / get / incr / expire

### 1.2 List 列表
- 底层：双向链表（快速头尾操作，中间查询慢）
- 用途：消息队列、排行榜、任务队列
- 常用命令：lpush / rpop / lrange

### 1.3 Set 集合
- 底层：哈希表，元素唯一、无序
- 用途：去重、交集/并集计算、好友共同关注、抽奖
- 常用命令：sadd / sinter / spop

### 1.4 Hash 哈希
- 底层：压缩列表/哈希表
- 用途：存储对象（用户信息、商品详情），单key存多字段，无需序列化整行
- 常用命令：hset / hget / hgetall

### 1.5 Sorted Set ZSet 有序集合
- 底层：跳表+哈希表，元素唯一，按score排序
- 用途：排行榜、延时队列、权重排序
- 常用命令：zadd / zrange / zrank

### 扩展高级数据结构
1. Bitmap：位图，签到、统计在线人数
2. HyperLogLog：基数统计，UV独立访客统计（误差0.81%）
3. Stream：持久化消息队列，支持消费组，替代List实现可靠MQ
4. Geo：地理位置坐标，附近人、距离计算

## 2. Redis 与 MySQL 双写数据一致性解决方案

### 方案1：先更新数据库，再删缓存（推荐标准方案）
流程：
1. 更新MySQL数据
2. 删除Redis对应缓存key
优点：操作简单，绝大多数场景保证最终一致；
问题：极小并发下短暂不一致，可通过延时双删优化。

### 方案2：延时双删（解决并发脏读）
1. 更新MySQL
2. 删除缓存
3. 休眠几百ms，再次删除缓存（清理期间并发写入的脏缓存）

### 方案3：先删缓存，再更新数据库
缺陷：高并发读场景极易产生脏缓存，不推荐生产使用。

### 方案4：更新缓存策略
频繁更新场景不推荐，频繁写会造成大量无效缓存更新；仅低频更新场景可用。

### 最终兜底方案：监听Binlog同步
通过Canal监听MySQL binlog，数据变更后自动更新/删除Redis缓存，解耦业务代码，一致性最强。

## 3. Redis 实现分布式锁 & Lua脚本优化

### 3.1 基础分布式锁（SETNX + EXPIRE 原子操作）
错误写法：setnx + expire 两条命令，宕机导致锁永久无法释放
正确原子命令：
```redis
SET lock_key unique_id NX EX 30
```
NX：不存在才设置；EX：过期时间，防止死锁；unique_id：唯一标识，防止误删别人的锁。

### 3.2 释放锁必须Lua脚本（保证原子判断+删除）
不能直接del，会误删其他线程持有的锁；Lua脚本保证判断、删除单原子执行：
```lua
if redis.call('get',KEYS[1]) == ARGV[1] then
   return redis.call('del',KEYS[1])
else
   return 0
end
```

### 3.3 分布式锁存在的问题
1. 锁超时，业务未执行完 → 看门狗续期（Redisson自动实现）
2. 主从切换，锁丢失 → RedLock红锁（多节点加锁）
3. 死锁风险 → 必须设置过期时间

## 4. Redis 高可用架构：哨兵Sentinel & Cluster集群

### 4.1 主从复制（基础）
- 主节点Master：写数据，同步数据给从节点Slave
- 从节点Slave：只读，数据副本，分担读压力
- 同步流程：全量RDB同步 + 增量缓冲复制积压日志

### 4.2 哨兵 Sentinel 架构（主从自动故障转移）

#### 核心作用
1. 监控：持续检测主、从节点在线状态
2. 通知：节点故障推送告警
3. 自动故障转移：主节点宕机，投票选出新主，修改其余从节点指向新主

#### 哨兵选举逻辑（面试重点）
1. **主观下线SDOWN**：单个哨兵判定主节点无响应，临时标记故障
2. **客观下线ODOWN**：超过半数哨兵统一判定故障，确认主节点宕机
3. 哨兵集群投票选举Leader哨兵，由Leader执行切换流程
4. 选举新主：优先级、复制偏移量、runid依次对比，选出最优从库升级为主库
5. 其余从节点重新同步新主，客户端自动切换新主地址

### 4.3 Cluster 分片集群（海量数据扩容）
1. 16384个哈希槽，每个节点分配部分槽位存储数据
2. 至少3主3从，主节点故障，从节点自动顶上
3. 哈希槽CRC16(key)%16384定位数据节点
4. 支持在线分片扩容、迁移槽位，横向扩展存储与并发

## 5. Redis 两大持久化机制 RDB & AOF

### 5.1 RDB 快照持久化
- 机制：定时fork子进程，全量dump内存数据到磁盘二进制文件
- 触发方式：save同步阻塞、bgsave后台异步、shutdown停机自动触发
- 优点：文件体积小、恢复速度快、适合冷备份
- 缺点：两次快照之间宕机，丢失中间全部数据，丢失风险高

### 5.2 AOF 日志持久化
- 机制：记录每一条写操作命令，追加写入日志文件
- 三种刷盘策略：
  1. always：每条命令落盘，安全，性能极差
  2. everysec：每秒刷盘，平衡性能与安全，默认配置
  3. no：交给操作系统调度，丢失数据风险高
- AOF重写rewrite：压缩日志，合并重复指令，减少文件体积
- 优点：数据丢失最多1秒，安全性高
- 缺点：文件体积大，故障恢复速度慢于RDB

### 混合持久化（Redis4.0+推荐）
重启恢复时先加载RDB快照，再追加AOF增量日志，兼顾恢复速度与数据安全。

## 6. 缓存三大问题：穿透、击穿、雪崩

### 6.1 缓存穿透
现象：查询Redis不存在、MySQL也不存在的数据（如恶意传非法ID），绕过缓存直接打满DB。
解决方案：
1. 空值缓存：查询不存在时缓存空对象，设置短过期时间
2. 布隆过滤器：将所有合法主键存入过滤器，非法请求直接拦截
3. 参数校验：接口提前过滤非法负数、超长ID

### 6.2 缓存击穿
现象：热点key过期瞬间，大量并发请求击穿到数据库。
解决方案：
1. 互斥锁：Redis分布式锁控制，同一时间仅一个线程查DB
2. 永不过期：热点key逻辑不设置过期时间，后台主动更新
3. 过期时间随机偏移，避免集中过期

### 6.3 缓存雪崩
现象：大量key同时过期，大量请求同时访问数据库，压垮DB。
解决方案：
1. 过期时间加随机值，打散过期时间
2. 集群高可用，避免Redis整体宕机
3. 多级缓存：本地Caffeine + Redis二级缓存
4. 限流、熔断降级，保护数据库

## 7. Redis 内存淘汰策略（maxmemory-policy）
当内存达到maxmemory上限，自动淘汰key，分三类：

1. **不淘汰策略 noeviction**
   写入直接报错，默认策略，不适合高并发业务。

2. **淘汰所有key（包含未设置过期）**
   - allkeys-lru：淘汰最近最少使用key（最常用）
   - allkeys-lfu：淘汰访问频率最低key
   - allkeys-random：随机删除任意key

3. **仅淘汰带过期时间的key**
   - volatile-lru：过期key中淘汰最少使用
   - volatile-lfu：过期key中淘汰低频访问
   - volatile-random：过期key随机删
   - volatile-ttl：优先删除剩余过期时间最短key

生产推荐：`allkeys-lru`，优先释放冷数据。

## 8. Redis 过期key删除策略

### 8.1 惰性删除（被动删除）
访问key时才判断是否过期，过期直接删除；
优点：无额外CPU消耗；缺点：大量冷过期key长期占用内存。

### 8.2 定期删除（主动清理）
Redis后台定时线程，随机抽取一批过期key批量删除；
优点：释放内存；缺点：单次扫描数量有限，无法清理全部过期key。

### 组合逻辑
惰性删除+定期删除双机制配合，平衡CPU与内存占用。

## 9. Redisson 分布式锁 看门狗（自动续期）

### 作用
业务执行时长超过锁过期时间时，自动延长锁有效期，防止业务未完成锁提前释放。

### 底层逻辑
1. 加锁成功后启动后台守护线程（看门狗）
2. 默认每隔 `锁过期时间/3` 自动续期30秒
3. 业务执行完成主动释放锁，看门狗线程停止
4. 节点宕机看门狗失效，锁到期自动释放，避免死锁

### 优势
原生封装，无需手动编写Lua续期脚本，解决锁超时业务未执行完毕问题。

## 10. Redis 秒杀场景落地方案

### 核心痛点
1. 大量并发请求压垮MySQL
2. 超卖问题（库存扣减并发冲突）
3. 无效流量拦截，减少DB访问

### Redis优化手段
1. 库存预加载进Redis，Lua脚本原子扣减库存，防止超卖
2. 分布式限流，Redis漏斗/令牌桶拦截超额流量
3. 消息队列异步下单，削峰填谷，同步转异步减轻DB压力
4. 商品热点缓存，避免重复查询商品基础信息
5. 布隆过滤器拦截重复下单用户

## 11. 完整秒杀业务流程
1. 活动预热：商品库存、信息加载至Redis，初始化库存key
2. 前端限流：按钮置灰、验证码拦截无效请求
3. 网关层限流：限制单用户访问频率
4. Redis层：Lua脚本原子判断库存、扣减，库存不足直接返回失败
5. 库存充足：发送下单消息至MQ，前端返回排队中
6. 消费端异步消费消息：创建订单、扣减MySQL真实库存、生成物流单
7. 定时任务补偿：处理消息丢失、订单超时未支付回滚库存
8. 兜底降级：流量超出阈值直接返回活动结束，保护数据库

## 12. 高频面试拓展题

### 面试1：Redis为什么单线程还快？
1. 纯内存操作，无磁盘IO开销
2. IO多路复用，单线程处理大量连接
3. 避免多线程上下文切换、锁竞争开销
4. 数据结构设计高效（SDS、跳表、压缩列表）

### 面试2：RDB和AOF如何选择？
冷备份、快速恢复选RDB；数据安全、高并发写选AOF；生产推荐混合持久化。

### 面试3：哨兵和Cluster集群区别？
哨兵仅做主从故障自动切换，无法扩容存储；Cluster分片集群支持数据分片横向扩容，海量数据场景使用。

### 面试4：Lua脚本优势？
多条命令原子执行，减少网络IO往返，保证分布式锁、库存扣减等操作无并发冲突。
