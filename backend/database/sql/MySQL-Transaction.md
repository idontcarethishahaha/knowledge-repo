## 一、数据库事务基础定义

## 二、事务四大特性 ACID（必背面试题）

### 1. Atomicity 原子性

- 含义：事务是最小执行单元，不可拆分，全部成功 / 全部失败。
- 底层实现：**Undo Log 回滚日志**
- 原理：执行增删改时，提前记录数据修改前的原始快照；事务执行失败、主动ROLLBACK时，读取Undo Log反向恢复数据。
- 举例：转账A扣钱、B加钱，A扣款成功后B更新失败，通过Undo Log把A的余额恢复。

### 2. Consistency 一致性

- 含义：事务执行前后，数据库始终处于合法合规状态，只会从一个合法状态切换到另一个合法状态。
- 合法约束包含：主键唯一、非空、外键、唯一索引、业务金额平衡等。
- 核心逻辑：原子性、隔离性、持久性，最终都是为了保障一致性。
- 举例：转账前后，A+B总金额不变，不会出现钱凭空消失/多出。

### 3. Isolation 隔离性

- 含义：并发执行的多个事务相互隔离，一个事务看不到其他事务未提交的脏数据。
- 底层实现：**行锁、表锁 + MVCC多版本并发控制**
- 作用：解决并发事务三大问题：脏读、不可重复读、幻读。

### 4. Durability 持久性

- 含义：事务执行`COMMIT`提交成功后，数据修改永久落地，服务器断电、宕机、数据库崩溃都不会丢失。
- 底层实现：**Redo Log 重做日志**
- 原理：提交事务时，先写入Redo Log持久化磁盘，再刷缓冲池数据；数据库崩溃重启后，扫描Redo Log恢复已提交事务。

## 三、并发事务三大问题（面试高频）

多事务同时读写同一份数据，会产生三类数据异常：

1. **Dirty Read 脏读**

   一个事务读取到了其他事务**未提交**的修改数据；若对方事务回滚，读到的数据就是无效脏数据。

2. **Non-Repeatable Read 不可重复读**

   同一事务内，**两次读取同一行数据**，中间被其他事务更新并提交，两次查询结果不一致。

   聚焦：**Update 更新** 带来的数据变化。

3. **Phantom Read 幻读**

   同一事务内，两次按条件范围查询，中间其他事务插入/删除数据并提交，第二次查询多/少了行数据，像出现幻觉。

   聚焦：**Insert / Delete** 带来的数据行数变化。

## 四、MySQL四大事务隔离级别（InnoDB）

隔离级别由低到高，隔离性越强，并发性能越差，锁竞争越严重：

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 实现机制 |
|--------|------|-----------|------|--------|
| Read Uncommitted 读未提交 | 允许 | 允许 | 允许 | 无MVCC，直接读原始数据 |
| Read Committed 读已提交（Oracle默认） | 禁止 | 允许 | 允许 | MVCC，每次读最新快照 |
| Repeatable Read 可重复读（MySQL InnoDB默认） | 禁止 | 禁止 | 允许（间隙锁解决） | MVCC，事务内统一快照 |
| Serializable 串行化 | 禁止 | 禁止 | 禁止 | 完全锁表，串行执行 |

### 级别详解

1. **读未提交 RU**

   性能最高，无任何隔离，业务几乎不会使用。

2. **读已提交 RC**

   每次查询生成新快照，能避免脏读，但同一事务多次查询结果可能变化。

3. **可重复读 RR（MySQL默认）**

   事务启动时生成统一快照，整个事务内重复读取数据不变；InnoDB通过**临键锁（Next-Key Lock）** 解决幻读问题。

4. **串行化 SERIALIZABLE**

   最高隔离级别，所有SELECT自动加共享读锁，读写互斥，并发极低，仅金融强一致性场景使用。

### 修改隔离级别命令

```sql
# 查看当前隔离级别
SELECT @@transaction_isolation;

# 会话级临时修改
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
# 全局永久修改（my.cnf配置文件也可配置）
SET GLOBAL TRANSACTION ISOLATION LEVEL REPEATABLE READ;
```

## 五、Undo Log 与 Redo Log 区别（深度面试题）

### 1. Undo Log（回滚日志）

- 作用：保障原子性、MVCC快照读
- 存储内容：数据修改前的原始旧版本
- 写入时机：修改数据前写入
- 生命周期：事务提交后不会立刻删除，MVCC需要旧版本快照，后台purge线程清理
- 用途：事务回滚、多版本快照读

### 2. Redo Log（重做日志）

- 作用：保障持久性，崩溃恢复
- 存储内容：数据修改后的新值、物理页修改记录
- 写入时机：事务提交前强制落盘
- 生命周期：数据刷入磁盘后可覆盖复用
- 用途：宕机重启恢复已提交事务

### 核心一句话区分

Undo Log存修改前数据，用来回滚；Redo Log存修改后数据，用来崩溃恢复。

## 六、MVCC 多版本并发控制（RR隔离核心）

### 1. MVCC作用

不加锁实现读写并发，读不阻塞写、写不阻塞读，大幅提升并发性能，仅在 `RC / RR` 隔离级别生效。

### 2. 底层隐藏字段

InnoDB每行数据自带3个隐藏列：

1. `DB_TRX_ID`：最后修改该行的事务ID
2. `DB_ROLL_PTR`：回滚指针，指向Undo Log里的旧版本数据，形成版本链
3. `DB_ROW_ID`：无主键时自动生成的隐藏主键

### 3. 快照读 vs 当前读

1. **快照读（普通SELECT，不加锁）**

   走MVCC，读取Undo Log历史快照，不加锁，并发友好。

2. **当前读（加锁读取）**

   `SELECT ... FOR UPDATE / LOCK IN SHARE MODE`、UPDATE、DELETE，读取磁盘最新数据，加行锁。

### 4. RR隔离快照机制

事务第一次快照读时生成Read View，后续所有查询复用同一个Read View，保证重复读取数据一致；RC每次查询都会生成新Read View。

## 七、高频面试场景题

### 面试题1：ACID分别靠什么日志实现？

- 原子性：Undo Log
- 持久性：Redo Log
- 隔离性：MVCC + 锁机制
- 一致性：原子、隔离、持久性共同保障

### 面试题2：MySQL默认隔离级别是什么？能解决幻读吗？

默认是 Repeatable Read 可重复读；

MVCC快照读仍会出现幻读现象，但InnoDB通过临键锁（Next-Key Lock）锁住区间，当前读场景下彻底杜绝幻读。

### 面试题3：为什么不推荐使用Serializable串行化？

所有查询加共享锁，读写互相阻塞，并发性能断崖式下跌，高并发系统极少使用。

### 面试题4：脏读、不可重复读、幻读怎么区分？

- 脏读：读到别人**未提交**的数据；
- 不可重复读：同一事务两次读同一行，中间被别人Update提交；
- 幻读：同一事务两次范围查询，中间被别人Insert/Delete提交。

### 面试题5：Undo Log事务提交后为什么不马上删除？

MVCC需要旧数据版本提供快照读，若有其他活跃事务还在读取旧快照，Undo记录必须保留，后台purge线程定期清理无用版本。

### 面试题6：什么是快照读？什么是当前读？日常SELECT属于哪一种？

不加锁普通SELECT是快照读，走MVCC；

UPDATE/DELETE/SELECT ... FOR UPDATE 属于当前读，加锁读取最新数据。

## 八、补充拓展：事务失效场景（开发踩坑）

1. 非InnoDB引擎（MyISAM不支持事务）；
2. 方法被`final`修饰、非public方法，AOP事务无法拦截；
3. 内部方法自调用，不走AOP代理，事务失效；
4. try-catch捕获异常未抛出，Spring无法触发回滚；
5. 抛出非RuntimeException，默认不会回滚；
6. SQL语句包含DDL（CREATE/DROP/ALTER），会自动提交前置事务。
