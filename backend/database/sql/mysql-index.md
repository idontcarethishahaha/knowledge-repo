# MySQL 索引原理、失效场景与面试高频题
## 一、索引基础概念
索引是一种特殊的数据结构，存储字段有序映射关系，核心目的是**避免全表扫描、加速查询/排序/分组/关联查询**。
### 索引核心作用
1. 加速数据检索，大幅减少磁盘IO，规避全表扫描
2. 优化排序、分组操作，利用索引天然有序性，避免文件排序 `Using filesort`
3. 约束数据唯一性：唯一索引、主键索引可限制列重复值
4. 加速表关联JOIN查询，关联字段建立索引可降低匹配开销

### MySQL 索引分类
1. **普通索引**：最基础索引，无唯一性限制，仅用于加速查询
2. **唯一索引**：列值全局唯一，允许一条NULL，查询、去重场景使用
3. **主键索引（聚簇索引）**：特殊唯一索引，不允许NULL，一张表仅一个，InnoDB主键即聚簇索引
4. **联合索引（复合索引）**：多个字段组合生成索引，遵循**最左前缀原则**
5. **全文索引**：针对长文本模糊检索，替代低效 `like %关键词%`
6. **覆盖索引**：查询所需全部字段都存在于索引树，无需回表，性能最优

### 索引过多带来的问题
1. **存储开销**：每个索引独立占用磁盘空间，大表会成倍增加存储占用
2. **写性能暴跌**：执行INSERT/UPDATE/DELETE时，所有关联索引都要同步更新，大量索引会加重写操作负担
3. **内存占用高**：频繁访问的索引会加载进缓冲池，索引过多易耗尽内存，触发频繁磁盘换页
4. **优化器选择困难**：索引数量过多，MySQL优化器需要遍历所有索引评估成本，增加解析耗时
5. **维护成本上升**：业务迭代、数据迁移时，多索引需要同步维护

### 适合创建索引 / 不适合创建索引的场景
#### ✅ 适合建索引的列
1. `WHERE` 条件频繁筛选的字段
2. `ORDER BY` / `GROUP BY` / `DISTINCT` 经常排序分组的字段
3. 多表JOIN关联的连接字段
4. 高区分度字段（唯一值占比高，如手机号、ID、用户名）

#### ❌ 不适合建索引的列
1. 表数据量极少（百条以内），全表扫描成本低于索引查询
2. 区分度极低的字段（如性别、状态，仅0/1两种值）
3. 更新频率极高的字段（频繁UPDATE会持续维护索引）
4. 超长字符串字段（索引占用空间过大，IO开销高）

### 索引设计选择原则
1. 优先选择区分度高的字段建立索引
2. 字符串索引使用前缀短索引，减少索引体积
3. 联合索引遵循最左前缀，等值条件放前、范围条件放最后
4. 按需创建，拒绝冗余索引，避免重复索引
5. 尽量设计覆盖索引，规避回表查询

## 二、聚簇索引 & 非聚簇索引（InnoDB核心）
### 1. 聚簇索引 Clustered Index
1. 索引B+树叶子节点**直接存储完整行数据**
2. 一张表只能存在1个聚簇索引，默认为主键
3. 无主键时，InnoDB会自动选取唯一非空索引作为聚簇索引；无唯一索引则生成隐藏ROW_ID作为聚簇索引
4. 数据在磁盘上按聚簇索引有序排列

### 2. 非聚簇索引（二级索引/辅助索引）
1. 索引叶子节点**不存完整数据**，仅存储主键ID
2. 一张表可以创建多个非聚簇索引
3. 查询流程：二级索引匹配到主键 → 拿着主键去聚簇索引查询完整数据，这个过程称为**回表**

### 回表查询 & 覆盖索引
#### 回表查询
InnoDB二级索引仅保存主键，检索到目标主键后，需要二次扫描聚簇索引获取完整行数据，两次磁盘IO会大幅降低查询性能。
示例：
```sql
-- name 为二级索引，查询*需要回表拿所有字段
SELECT * FROM user WHERE name = "test";
```

#### 覆盖索引（优化回表的最优解）
将查询需要的所有字段全部加入联合索引，查询时仅通过二级索引就能拿到全部数据，**完全跳过回表**。
示例：
```sql
-- 创建联合索引(name, id, age)
CREATE INDEX idx_name_age ON user(name, age);
-- 查询字段全部在索引内，触发覆盖索引，Extra显示Using index
SELECT id, age FROM user WHERE name = "test";
```

## 三、联合索引 & 最左前缀原则（面试高频）
### 1. 最左前缀底层逻辑
联合索引 `(A,B,C)` 的B+树排序规则：
1. 优先按字段A全局排序
2. A值相同时，再按B排序
3. A、B值均相同时，再按C排序

核心结论：B、C仅在前置字段等值匹配的局部区间有序，全局无序。
- `WHERE A=1`：能命中索引
- `WHERE A=1 AND B=2`：能命中索引
- `WHERE A=1 AND B=2 AND C=3`：完整命中索引
- `WHERE B=2` / `WHERE C=3`：跳过最左列，索引失效

### 2. 范围查询截断索引（必考面试题）
规则：**联合索引中，范围条件后的所有字段，无法走索引**
示例：索引 `(A,B,C)`
```sql
-- A等值、B范围，C失效
SELECT * FROM table WHERE A=1 AND B>10 AND C=2;
```
原因：B执行范围匹配后，B区间内的C不再有序，无法通过索引快速定位C，C字段失效。

### 3. 等值+ORDER BY 索引排序优化
场景1：等值字段在前，排序字段在索引末尾
```sql
-- 索引(A,B,C)
SELECT * FROM table WHERE A=1 AND B=2 ORDER BY C;
```
A、B全部等值匹配，索引内C天然有序，无需额外文件排序，`Extra` 无 `Using filesort`。

场景2：中间存在范围条件
```sql
SELECT * FROM table WHERE A=1 AND B>10 ORDER BY C;
```
B为范围条件，C失去有序性，MySQL需要额外排序，出现 `Using filesort`。

### 4. 联合索引核心总结
1. 必须从最左侧字段依次使用，跳过前置列直接查询右侧字段 → 索引失效
2. 范围条件会截断索引，范围字段后方所有列无法使用索引
3. 等值条件放联合索引左侧，排序字段放末尾，可规避文件排序
4. 按需设计覆盖索引，消除回表开销

## 四、索引失效全场景（高频面试）
1. **违反最左前缀原则**：跳过联合索引左侧字段直接查询右侧
2. **索引列参与函数/四则运算**
```sql
-- 失效：索引列使用函数
SELECT * FROM user WHERE SUBSTR(name,1,2) = "ab";
-- 失效：索引列参与运算
SELECT * FROM user WHERE age + 1 = 18;
```
3. **隐式类型转换**
索引字段varchar，查询条件传入数字，MySQL自动转换字段类型，索引失效
```sql
-- name VARCHAR，不加引号触发隐式转换，索引失效
SELECT * FROM user WHERE name = 123;
```
4. **模糊匹配通配符前置**
`like '%关键词'` 左侧带%，无法走索引；`like '关键词%'` 后缀通配符可正常命中
```sql
-- 失效
SELECT * FROM user WHERE name LIKE "%张三";
-- 生效
SELECT * FROM user WHERE name LIKE "张三%";
```
5. **不等、非空、not in 条件**
`!=`、`IS NOT NULL`、`NOT IN` 会导致索引失效，触发全表扫描
6. **OR 查询单侧无索引**
`WHERE A=1 OR B=2`，A有索引、B无索引，整条SQL索引失效
7. **范围条件截断索引**
联合索引中范围字段后的列全部失效

## 五、EXPLAIN 执行计划分析索引使用
### 使用方式
在任意SELECT语句前增加EXPLAIN关键字，查看SQL执行计划
```sql
EXPLAIN SELECT id,age FROM user WHERE name = "张三";
```

### 核心观测字段
1. **type（访问类型，优先级最高）**
性能从优到劣排序：`const > eq_ref > ref > range > index > ALL`
- `ALL`：全表扫描，索引完全失效，必须优化
- `range`：范围索引，如>、<、between、like前缀匹配
- `ref`：普通等值索引匹配，日常查询主流类型
- `eq_ref`：主键/唯一索引等值匹配，性能最优
- `const`：常量匹配，仅返回单行数据

2. **key**
- 显示索引名称：成功命中对应索引
- `NULL`：未使用任何索引，全表扫描

3. **rows**
预估扫描行数，数值越大性能越差，代表索引未有效过滤数据

4. **Extra（额外信息，优化重点）**
- `Using index`：触发覆盖索引，无回表，最优情况
- `Using filesort`：无法利用索引排序，需要磁盘文件排序，必须优化
- `Using temporary`：创建临时表存储中间数据，多出现于GROUP BY无索引场景
- `Using where`：服务器层过滤数据，索引仅过滤部分数据

## 六、B树 vs B+树（数据库索引底层）
### 结构差异
1. **B树**
   - 所有节点（根、分支、叶子）均存储数据行指针
   - 一条查询可能在中间分支节点直接结束，查询IO次数不稳定
   - 叶子节点无有序链表，范围查询需要递归中序遍历

2. **B+树（MySQL默认索引结构）**
   - 仅叶子节点存储数据指针，内部节点只存索引键值，树高度更低、IO更少
   - 所有叶子节点通过双向指针串联，天然有序链表
   - 所有查询必须走到叶子节点，查询IO次数稳定

### B+树作为数据库索引的优势
1. **磁盘IO更少**：内部节点无数据，单页可存放更多索引键，树更矮，减少磁盘寻道次数
2. **范围查询极快**：叶子节点有序链表，区间扫描直接顺序遍历，无需回溯
3. **查询性能稳定**：无论等值/范围查询，都需要走到叶子节点，性能波动小
4. **空间利用率更高**：内部节点轻量化，缓存命中率更高

## 七、高频面试场景题
### 场景1
表user，联合索引 `(phone, age)`，以下SQL哪些能走索引？
1. `WHERE phone = "13800001111"` → 命中
2. `WHERE age = 18` → 失效，跳过最左列phone
3. `WHERE phone = "13800001111" AND age > 18` → 两列均命中
4. `WHERE age > 18 AND phone = "13800001111"` → 命中，优化器自动调整条件顺序

### 场景2
索引 `(id, name, create_time)`，SQL：
```sql
SELECT name FROM user WHERE id=1 AND create_time > "2026-01-01";
```
问题：是否回表？
答：索引包含id、name、create_time，查询仅需要name，触发覆盖索引，无回表。

### 场景3
索引 `(a,b,c)`
```sql
SELECT * FROM t WHERE a=1 AND b>5 ORDER BY c;
```
问题：是否产生filesort？
答：会。b是范围条件，c在b区间内无序，MySQL需要额外排序。

### 场景4
varchar字段user_name建立索引，查询：
```sql
SELECT * FROM t WHERE user_name = 123456;
```
问题：索引是否失效？
答：失效。字符串字段匹配数字，触发隐式类型转换，索引列被函数处理。

## 八、补充：雪花算法（拓展）
### 1. 雪花算法作用
分布式场景下生成全局唯一、有序的64位Long类型ID，替代自增主键，解决多库分表ID重复问题。

### 2. 64位结构（标准）
1. 第1位：符号位，固定0，保证正数
2. 41位：时间戳（毫秒级），可使用69年
3. 10位：机器ID（5位机房ID + 5位机器ID），最多部署1024台节点
4. 12位：序列号，同一毫秒内自增，单节点每毫秒最多生成4096个ID

### 3. 优缺点
✅ 优点：
1. ID全局唯一，天然时间有序，适合作为InnoDB聚簇主键
2. 高性能，纯内存计算，无IO损耗
3. 可拆分机器位，支持分布式集群部署

❌ 缺点：
1. 依赖服务器系统时间，时钟回拨会生成重复ID
2. ID长度较长，占用存储空间高于自增INT主键
3. 无内置业务含义，无法从ID解析业务信息

### 4. 适配MySQL索引优势
雪花ID有序递增，作为聚簇主键时，新数据持续追加到索引尾部，不会频繁产生页分裂，大幅降低插入写性能损耗。
