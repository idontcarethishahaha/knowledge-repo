### 1. 链地址法

**最常用、最推荐**

+ 哈希表的每个位置（桶）不直接存储值，而是指向一个链表（或红黑树）；
+ 当冲突发生时，将冲突的键值对插入到该位置对应的链表末尾；
+ 若链表过长（如超过 8 个元素），可优化为红黑树，提升查询效率。
  
**优点**
+ 实现简单，处理冲突高效；
+ 不会出现 “聚集” 问题；
+ 空间利用率高（链表动态扩展，无需预分配大量空间）；
+ 删除操作方便（只需操作链表节点）。
  
**缺点**
+ 需要额外的指针存储开销（链表节点的 next 指针）；
+ 链表过长时，查询效率从 O (1) 退化为 O (n)（优化为红黑树后是 O (logn)）。

### 2. 开放寻址法

冲突发生时，不在哈希表外开辟空间，而是在哈希表内部按某种规则 “探查” 下一个空闲的地址，直到找到为止。核心是 “探查序列”（寻找空闲地址的规则）。
常见的探查方式
##### (1) 线性探查（Linear Probing）
规则：冲突时，依次检查下一个地址 index+1, index+2, ..., 哈希表长度-1, 0, 1...，直到找到空闲地址。
优点：实现最简单；
缺点：易出现 “一次聚集”（连续的地址被占用，导致后续探查需要走很远），查询 / 插入效率大幅下降。

##### (2) 二次探查（Quadratic Probing）
规则：冲突时，按平方步长探查 index+1², index+2², index-1², index-2²...（避免连续步长）。
优点：缓解 “一次聚集”；
缺点：可能出现 “二次聚集”（相同哈希值的键探查序列相同），且不一定能找到空闲地址。
##### (3) 双重哈希（Double Hashing）
规则：用第二个哈希函数计算探查步长 step = hash2(key)（step 不能为 0），冲突时探查 index+step, index+2*step...。
优点：解决 “聚集” 问题，是开放寻址法中性能最好的；
缺点：需要设计两个优质的哈希函数（增加复杂度）。

### 3. 公共溢出区法 （简单小众）

+ 把哈希表分为「基本表」和「溢出表」
+ 所有能正常哈希的 key 存入基本表
+ 冲突的 key 全部存入溢出表
  
优点：实现极简单，**适合冲突少**的场景；
缺点：溢出表满后，效率急剧下降，**仅适用于小规模数据**

##  冲突的 “预防”

除了冲突发生后的解决方法，提前预防冲突更关键：

+ 设计优质哈希函数：让 key 的哈希地址均匀分布（比如避免简单的取模，结合位运算、乘法等）；
+ 设置合理的负载因子：**负载因子 = 元素个数 / 哈希表大小** ，超过阈值（比如 0.75）就「**扩容**」哈希表（比如 Java HashMap 扩容为原来的 2 倍），**减少冲突概率**；
+ 选择合适的哈希表大小：尽量选质数（比如 11、17），减少取模时的冲突（比如 key 是偶数，哈希表大小为偶数会导致地址都是偶数，聚集严重）。


## 链地址法：链表 + 红黑树

### 为什么需要 “链表 + 红黑树” 混合结构？

Java HashMap 底层是链地址法的典型实现：数组（桶）+ 链表是基础，但链表有个致命问题：

**链表的查询 / 插入 / 删除时间复杂度是 O(n)**

当单个桶的链表过长时，HashMap 的性能会急剧下降（比如链表长度 100，查一个元素要遍历 100 次）。

而 **红黑树是一种「自平衡的二叉查找树」**，查询 / 插入 / 删除的 **时间复杂度是 O(logn)**，能大幅优化长链表的性能。

因此 Java 8 引入了 “链表转红黑树” 的设计：**短链表用链表（开销小），长链表转红黑树（效率高），兼顾空间和时间效率**

### 链表转红黑树的「严格条件」

Java HashMap 转树需要满足两个核心条件：

<u>**条件 1：单个桶的链表长度 ≥ 8**</u>
这是触发转树的 “长度阈值”，Java 官方基于「泊松分布」统计：链表长度达到 8 的概率仅为百万分之几（几乎是极端场景），因此选 8 作为转树阈值（日常场景链表长度都很短，用链表足够）

<u>**条件 2：HashMap 的数组（桶数组）容量 ≥ 64**</u>
如果数组容量 < 64，即使链表长度≥8，也不会转树，而是先触发「数组扩容」（扩容后元素会重新哈希，分散到更多桶中，链表长度自然缩短）。
👉 **原因：数组容量小时，扩容的成本远低于转树，且扩容能从根源上减少链表长度，比转树更高效**

### 链表转红黑树的具体过程

**步骤 1：节点类型转换**
链表的节点是 HashMap.Node 类型（包含 key、value、next 指针）；
转树时，先把所有 Node 节点转为 HashMap.TreeNode 类型（TreeNode 继承自 Node，新增了红黑树的核心属性：parent（父节点）、left/right（左右子节点）、red（节点颜色）等）。

**步骤 2：构建红黑树**
将转换后的 TreeNode 节点，按照红黑树的规则（保持自平衡）构建树：
以链表的第一个节点为根节点；
依次将剩余节点插入红黑树（按 key 的哈希值比较大小，确定左 / 右子节点）；
插入后调整树的结构（变色、左旋 / 右旋），满足红黑树的 5 条核心特性：
每个节点非红即黑；
根节点是黑色；
所有叶子节点（NIL）是黑色；
红色节点的子节点必须是黑色；
从任意节点到其所有叶子节点的路径，包含相同数量的黑色节点。

**步骤 3：替换桶的存储结构**
桶中原本指向链表头节点的引用，改为指向红黑树的根节点，后续对该桶的操作（查 / 增 / 删）都按红黑树逻辑执行。

### 红黑树转回链表的条件

为了**避免 “少量元素时红黑树的开销大于收益”**（红黑树有旋转 / 变色的额外开销），当桶中的元素数量减少到一定阈值时，会把红黑树转回链表：

**触发条件：桶中的元素数量 ≤ 6（转回阈值）；**
为什么是 6？：和转树阈值 8 之间留了「2 的间隔」，**避免 “链表长度在 8 左右波动时，频繁在链表和红黑树之间切换”**（比如长度 7 时不转，防止来回转换的性能损耗）。


### 总结

+ 转树的核心目的：优化长链表的查询效率（O (n)→O (logn)）；
+ 转树条件：① 单个桶链表长度≥8；② 数组容量≥64（缺一不可）；
+ 转回链表条件：桶中元素数量≤6（避免频繁切换）；
+ 8/6 的阈值基于概率统计，优先扩容再转树，兼顾性能和开销。
+ Java HashMap 是 **按需优化** —— 短链表用链表（轻量），极端长链表转红黑树（高效），数组容量小时先扩容分散元素，不轻易转树。
