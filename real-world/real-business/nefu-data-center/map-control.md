1. **介绍一下这个项目？**
> 我在实习期间，参与开发了病虫害数字化管控平台，主要面向国家林业部门使用。平台基于 GIS 技术，形成了病虫害监测、预警、核查、取样、防治的任务闭环，实现从数据采集、可视化分析到决策处理的全流程数字化，帮助林业部门更精准更及时地防控病虫害

2. **为什么做这个项目，业务背景？**
> 传统林业病虫害监测依赖人工巡查，效率低、数据分散、空间定位不准，难以及时预警和不能统一调度。
国家林业部门需要一套数字化、可视化、空间化的管控平台，把监测数据、地理信息、核查任务、取样任务、防治任务统一管理，形成 “监测 — 预警 — 处置” 的任务闭环，辅助决策处理和提升防控效率

3. **你在这个项目中负责什么/做了什么？**
我主要负责前端功能开发，具体有：
- 使用 Echarts 实现样地检测数据、病虫害信息等多维度数据可视化；
接入大疆无人机 上云API，实现远程语音喊话功能；
对接天地图 API，完成坐标系转换与符号化配置，展示各区域病虫害信息；
基于自研地形配准算法，将高程数据映射到 Cesium 三维球上，实现病虫害点位三维展示；

Cesium 默认：WGS84 (EPSG:4326)
天地图默认：CGCS2000 (EPSG:4490)

- 一开始我用的是相机高度判断：
相机高度大于 1500 米时开启聚合，小于 1500 米关闭聚合，直接展示所有点位。
但上线后发现有问题：我们林区地形起伏大，相机在山顶时高度很高，但视野范围其实很小，会出现明明已经很近了还在聚合的情况。
- 所以我做了第一次优化：
不再直接用相机高度，而是从相机位置垂直向下发射一条射线，与地球地形求交点，算出相机到地面的真实距离。
用这个距地高度作为聚合阈值，大于 800 米开启聚合，小于 800 米不聚合，这样地形高低就不会影响判断了。
- 后来我又进一步优化，改用更通用、体验更稳定的屏幕像素距离聚合：
遍历所有点位，转成屏幕坐标，判断两点之间的像素距离，
如果小于 24 像素，就把它们合并成一个聚合点，显示数量。
最后为了保证在近视图下一定不聚合，我加了一层兜底逻辑：
只要距地高度低于 500 米，就强制关闭聚合，只展示真实点位。
最终方案就是：
> 距地高度 > 500m 时，启用像素距离聚合（24px 阈值）；低于 500m 直接展示全部点位。
> 
> 聚合后，聚合的标志是一个点位，然后点击该聚合标志点位，可以flyto上方500米

cesium默认高度是椭球高

4. **为什么做无人机喊话 / 三维展示这些功能？怎么想到的？**
无人机喊话：林业区域范围大、人员难到达，远程喊话能快速通知现场人员、警示违规作业、配合巡查，提升应急处置效率；
三维地形展示：传统二维地图无法体现山林高程起伏，病虫害实际发生位置与地形高度密切相关，三维展示能让决策人员更直观判断分布、坡度、交通难度，提高防控精准度；
这些需求都是业务场景驱动的

5. **技术栈横向对比：为什么选 Vue3/Cesium/Echarts/ 天地图？**
Vue3：相比 Vue2 组合式 API 更适合复杂逻辑拆分，响应式更稳定，配合生态成熟，适合后台管理系统；
Cesium：对比 Leaflet、Mapbox，Cesium 在三维地形、全球渲染、海量点位方面优势明显，适合林业大范围三维场景；
Echarts：图表类型丰富、配置灵活、性能好，适合林业多维度数据展示；
天地图：国家官方地理底图，坐标体系规范、精度高，适合政务类林业项目；
TailwindCSS：原子化样式开发快、一致性强，适合快速搭建后台页面。
整体选择都是围绕GIS 场景复杂、数据量大、政务系统稳定这几个核心点。

6. **场景题：二维地图 vs 三维地图，你怎么选？**
若只做区域概览、数据统计、简单点位查看，用二维地图（天地图）足够，加载更快、操作更轻；
若需要展示山林地形、高程差异、真实空间分布、巡检路径，必须用三维 Cesium，因为地形信息对林业防控决策非常关键。
我在项目里是二三维结合：基础信息用二维，深度研判用三维。

7. **项目难点是什么？**
多坐标系转换问题
天地图、无人机、业务数据坐标系不一致，直接叠加会出现点位偏移，需要统一转换并校准。
Cesium 三维点位与地形精准匹配
山林高程变化大，直接贴地会出现点位漂浮或钻入地下，需要自己实现地形配准 + 高程采样逻辑。
大量病虫害点位渲染卡顿
点位多时地图卡顿，需要做聚合、层级显隐、渲染优化。
无人机 API 异步控制逻辑复杂
指令下发、状态回调、异常重发需要严谨的状态管理。
