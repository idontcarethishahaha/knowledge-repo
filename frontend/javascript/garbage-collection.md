## 浏览器垃圾回收

浏览器垃圾回收（Garbage Collection，GC） 是浏览器 JavaScript 引擎自动管理内存的核心机制，简单来说就是 **引擎自动找出 <u>「不再使用的内存」</u> 并释放，避免内存泄漏**

- **「不再被使用」** 的定义：没有任何可达的引用指向该对象（即从根节点（如全局对象、调用栈中的变量）出发，无法找到这个对象

<hr/>

### 浏览器主流垃圾回收算法

#### 1. 标记 - 清除：最核心的算法

##### 阶段 1：标记（Mark）
从「根节点」（全局对象 window、当前调用栈中的变量、事件循环队列中的回调等）出发，**遍历所有可达的对象，给这些对象打上「存活」标记**

> 示例：如果 let obj = { name: '张三' }，obj 是调用栈中的变量（根节点），则 { name: '张三' } 会被标记为存活。

##### 阶段 2：清除（Sweep）

**遍历堆内存中所有对象，清除（释放）没有被标记为「存活」的对象，回收其占用的内存**


#### 优化：标记 - 整理（Mark-and-Compact）

- **标记 - 清除的问题**：回收后内存会产生 **「碎片」**（不连续的空闲内存），影响大对象的分配

<u>**标记 - 整理在「标记」后，先将存活对象移动到内存的一端，再清除另一端的所有内存，避免碎片**</u>

#### 2. 引用计数：已被淘汰的算法

- 给每个对象维护一个 **「引用计数器」**，记录指向该对象的引用数量；
- 引用数为 0 时，立即回收该对象；
> 致命缺陷：**无法解决「循环引用」问题**（导致内存泄漏）

<hr/>

### V8 引擎的垃圾回收优化（重点）

Chrome/V8 针对「标记 - 清除」做了分层优化（因为 **JS 堆内存分为「新生代」和「老生代」**）：

1. **新生代：存放短期存活的对象**（如临时变量）

- 内存空间小（通常 1-8MB），回收频率高；
- 使用 **「Scavenge 算法」**：将新生代内存分为 **From 区**（使用中）、**To 区**（空闲）；
   - 标记 From 区的存活对象，复制到 To 区并整理内存；
   - 交换 From/To 区，释放原 From 区内存（直接互换 From/To 的 “身份”）
若对象多次复制仍存活（超过阈值），则晋升到「老生代」。

1. **老生代：存放长期存活的对象**（如全局变量、DOM 绑定对象）

- 内存空间大，回收频率低；
- 主要使用「标记 - 清除 + 标记 - 整理」，并优化为「增量标记 + 并发回收」：

- **增量标记**：
  - 传统标记 - 清除是 “一次性遍历全堆”，会阻塞 JS 执行（称为 “GC 停顿”）；
  - 增量标记把整个标记过程拆成多个小步骤，穿插在 JS 执行间隙完成，大幅降低单次停顿时间（从几百毫秒降到几十毫秒）
- **并发回收**：垃圾回收线程与 JS 执行线程并行运行（**仅标记 / 整理阶段**），进一步 **减少卡顿**

<hr/>

### 垃圾回收的触发时机

**浏览器不会实时回收垃圾，而是在特定时机触发**

1. **新生代内存不足**：From 区占满时，触发 Scavenge 回收；
2. **老生代内存不足**：**达到内存阈值**（如堆内存占用超过 70%），触发标记 - 清除 / 整理；
3. **浏览器空闲时**：事件循环的空闲阶段（Idle Callback），执行低优先级的回收；
4. **页面卸载时**：关闭标签页 / 刷新页面，回收该页面的所有内存。
